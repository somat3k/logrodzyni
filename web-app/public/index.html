💡 JWT Tips

  • Never expose your signing secret in client-side code or logs.
  • Always set an expiry (exp). Short-lived tokens (≤8h) reduce blast radius on theft.
  • Store tokens in httpOnly cookies or memory — not localStorage — to prevent XSS exfil.
  • Always verify the alg header server-side; reject alg: none unconditionally.
  • Include the minimum required claims; never embed sensitive PII in the payload (it's only base64, not encrypted).
  • Use short-form claim names (sub, iss, aud) per RFC 7519 for interoperability.
  • Rotate secrets regularly and keep an old-key grace period for in-flight tokens.

🗺 Implementation Plan Full delivery roadmap

Phase 1 Scope & Constraints

  • Target: Linux-first; cross-platform C++ via Boost.Asio; Node.js for web app
  • Concurrent users: up to 10 000 sessions per relay node (epoll I/O model)
  • Latency goal: < 5 ms p99 added overhead per hop
  • Compliance: operators must confirm proxy-use is lawful in their jurisdiction

Phase 2 Repository & Delivery Structure

  • Monorepo: proxy-core/, web-app/, masking-client/, shared-security/, deploy/, docs/
  • CI: GitHub Actions — build, unit tests, CodeQL, Trivy, npm audit
  • CD: GitHub Pages (SPA dashboard), Docker Hub (proxy-core & web-app images)
  • Environment profiles: dev (sqlite+local), stage (docker-compose), prod (Kubernetes/Nomad)

Phase 3 Premium Architecture

  • Layered circuit: Ingress Gateway → Relay Nodes → Egress Node → Policy Engine
  • mTLS with short-lived leaf certs between every hop (shared-security library)
  • Observability: Prometheus metrics, structured JSON logs, OpenTelemetry traces
  • Multi-tenant: per-tenant routing tables, quota policies, isolated egress pools

Phase 4 C++ Proxy Core

  • Boost.Asio async I/O with thread-pool (hardware_concurrency threads)
  • Protocols: SOCKS5 (RFC 1928), HTTP CONNECT — both on configurable ports
  • Bidirectional async relay chains — no blocking reads in hot path
  • Policy engine: CIDR ACL, glob destination matching, per-IP rate limiter (token bucket)
  • Resilience: exponential backoff reconnect, relay failover, graceful shutdown

Phase 5 Secure Network Layer

  • E2E encryption: TLS 1.3-only, AES-256-GCM data plane, ECDHE key exchange
  • Cert lifecycle: CA signing, SAN-bearing leaf certs, automated rotation + revocation
  • Network hardening: iptables isolation, minimal port exposure, non-root containers
  • DDoS mitigation: nginx rate limiting, connection throttling at ingress

Phase 6 Web Control Plane

  • Auth: password (bcrypt), SHA-256 key, MetaMask wallet login, guest viewer
  • RBAC: viewer < operator < admin — enforced per endpoint
  • Persistent SQLite DB: users, nodes, sessions, policies, audit log
  • JWT Creator tool with best-practice tips and token decoder/verifier
  • GitHub Pages SPA deployment with automated workflow

Phase 7 Masking Client

  • Lightweight C++ client with INI-based profile switching (standard / strict / region)
  • iptables kill-switch chain — auto-cleaned on exit or crash
  • Exponential backoff reconnect (2ⁿ seconds, cap 60 s)
  • Low footprint: Boost.Asio async event loop, < 10 MB RSS

Phase 8–11 Security, Testing, Ops & Docs

  • Threat model: MITM, replay, credential theft, traffic correlation mitigations documented
  • Test matrix: unit (Catch2 / Jest), integration (supertest), performance (wrk/k6), chaos
  • Deployment: reproducible Docker builds, canary roll-out, automated rollback triggers
  • Docs: architecture, API spec, security model, operator playbook, onboarding guide
