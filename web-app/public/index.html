Security Tips
  • Never expose your signing secret in client-side code or logs.
  • Always set an expiry (exp). Short-lived tokens ≤8h reduce blast radius.
  • Store tokens in httpOnly cookies or memory — never localStorage.
  • Always verify the alg header server-side; reject alg:none unconditionally.
  • Include minimum required claims; never embed sensitive PII in payload.
  • Rotate secrets regularly and keep an old-key grace period for in-flight tokens.

Implementation Plan

Full delivery roadmap.

Phase 1 Scope & Constraints

  • Linux-first; cross-platform C++ via Boost.Asio; Node.js for web app
  • Up to 10 000 concurrent sessions per relay node (epoll I/O model)
  • Goal: < 5ms p99 added overhead per hop; AES-256-GCM data plane

Phase 2 Repository & Delivery

  • Monorepo: proxy-core/, web-app/, masking-client/, shared-security/, deploy/, docs/
  • CI: GitHub Actions — build, unit tests, CodeQL, Trivy, npm audit
  • CD: Docker Hub images, optional Kubernetes/Nomad deployment

Phase 3 Architecture

  • Layered circuit: Ingress Gateway → Relay Nodes → Egress → Policy Engine
  • mTLS with short-lived leaf certs between every hop
  • Observability: Prometheus metrics, structured JSON logs, OpenTelemetry traces

Phase 4–8 Proxy Core, Security, Testing, Ops, Docs

  • Boost.Asio async I/O; SOCKS5 + HTTP CONNECT; token-bucket rate limiter
  • TLS 1.3-only; ECDHE key exchange; automated cert rotation
  • Test matrix: unit (Catch2/Jest), integration (supertest), perf (wrk/k6)
